## 1. What this slice is (and what it is deliberately not)

This is a **generic CLI capability**. The tool operates on whatever repository it
is pointed at; this repo is one such repository (used for validation, §7). No
design choice here is motivated by "this repo happens to have problem X."

The umbrella's conformance arm (§13.1) has a strength hierarchy; this slice builds
**rung-2 for docs**: a doc that a deterministic generator can re-emit from code is
checked by **regen-and-diff** — regenerate, diff against the committed copy,
hard-fail on mismatch, auto-fix on demand. The existing `cli-reference` drift check
is exactly this pattern but **hand-wired for one doc**. This slice generalizes it
into a **registry + one generic `doc check` gate**.

**Why a registry, not two hand-wired CI steps (the real justification).** For *this
repo's* current N=2 derivable docs, two `--check` CI steps would do — so "prove N>1"
is a weak motive (and is honestly demoted here). The registry earns its keep on
**adopter extensibility**: an adopter who runs `init` cannot edit this repo's CI
steps, and must not have to fork the bundled workflow template to guard *their* own
derivable docs. A declared `.harness/derived-docs.yaml` is the only surface that
lets them add a derivable doc with config, not a code fork. That — not in-repo doc
count — is why the indirection exists. (If no adopter will register a derivable doc
in the v0.1/v0.2 window, the registry would be premature and two hand-wired steps
would be the YAGNI-correct call; we judge adopter-facing regen-diff to be a
load-bearing part of the bundled gate, so we build it now.)

**In scope — the `derivable doc ↔ generator` link, end to end:**

1. A **registry** (`.harness/derived-docs.yaml`): a declared list of
   `path → command`, where `command` re-emits the doc's canonical content to
   stdout. *Declared, not detected* — a doc is derivable because the author
   registered a generator for it (umbrella §5: "which checker a link uses is
   declared").
2. A **registry loader** that borrows the **YAML *shape*** of
   `core/source_scope.py` (a top-level mapping key holding the payload) but its
   **fail-*closed* error handling** from the slice-1 decision path
   (`core/decisions.py`'s `RecordError` → `decision_check` early-returns
   `EXIT_NO_CONFIG`) — **NOT** `source_scope`'s deliberately fail-*open*
   swallow-and-default. A malformed registry must block (exit `3`), never silently
   default to "no docs".
3. A generic **`doc check [--fix] [--json]`** command. The tool owns the
   regen/diff/fix/exit/json semantics uniformly; the generator is a dumb emitter.
4. **Self-host wiring**: replace this repo's bare `gen_cli_reference --check` CI
   step with `super-harness doc check`, and mark it required.
5. **Adopter wiring**: a `doc check` step in the bundled `init` workflow template.
6. **Migration of the one existing generator** (`gen_cli_reference`) into the
   registry, and a **second derivable doc** — `state-machine.md`, generated by
   *calling the live state machine* (`compute_target_state` over every
   state×event pair, §5.2) so the doc is total and genuinely `f(code)`. It is a
   real reference page (the full reachability table) and the engine's second real
   consumer; it is **not** a contrived instance minted only to make N>1.

**Explicitly NOT in scope** (separate functional scopes, §8 — deferred *by
boundary*, not laziness):

- The **prose** half of the doc leg: `@decision:` anchors *inside docs*. This
  splits into two separable mechanisms (do not conflate them):
  - *Referential integrity* — "a doc's `@decision:<id>` names a ratified decision"
    — is **teeth-free**: it is exactly slice-1's dangling-up check, one file-class
    over. It has genuine standalone value (slice-1 shipped dangling-only precisely
    *because* it killed the 34 floating labels), so it is **not** valueless — it is
    deferred for **sequencing**, not because it does nothing.
  - The high-value *"anchored thing changed → force re-review"* proxy (umbrella §5
    proxy checker / §7.2) **does** need change-detection, which does not exist yet,
    and rides with the deferred integrity-lock / "teeth" slice.
  We ship the derivable leg first (highest mechanical value, auto-fixable) and take
  the whole prose half together with the teeth, rather than landing a half-prose
  mechanism now. (Brainstorm 2026-06-11; the user chose "derivable-only" with the
  prose referential-integrity option explicitly on the table.)
- The **sedimentation arm** (umbrella §13.1 ARM-2: decaying reusable knowledge) —
  a different lifecycle entirely; out of scope, not a TODO here.
- Tying a derived doc to the **decision** it serves (umbrella §13.4). regen-diff's
  verdict is **mechanically complete without it** — the ground truth is
  `generator(code)`, and no checker would read a `decision:` field today. So the
  §13.4 decision-tie is *additive surfacing* ("this drifted doc serves D"),
  deferred until a consumer reads it — **not** omitted as worthless. (Adding it now
  would re-commit the slice-1 mistake of shipping config before a reader exists:
  `source-paths.yaml` sat unread until slice-1's `decision check` finally consumed
  it.)

This slice introduces **only** the `derived-docs.yaml` registry and the `doc check`
gate. It does not touch the `decision` group, the `@decision:` anchor scanner, or
any lifecycle plumbing.

## 2. The registry

### 2.1 Storage & format

- **One file**: `.harness/derived-docs.yaml` (under the tracked `.harness/`
  config dir, beside `source-paths.yaml`).
- **Shape** (verified against the `source-paths.yaml` precedent — a top-level
  mapping key holding the payload):

```yaml
derived_docs:
  - path: docs/cli-reference.md
    command: python -m scripts.gen_cli_reference --emit
  - path: docs/state-machine.md
    command: python -m scripts.gen_state_machine --emit
```

- `path` is **repo-relative** (resolved against repo root, like every other
  super-harness path input) and MUST stay **inside the repo** — see §2.2 (this is
  load-bearing because `--fix` *writes* to `path`).
- `command` is the generator invocation. Its contract (§3): print the doc's
  canonical content to **stdout**, exit `0`.
- **Pure `path → command` map. No `decision:` field** (§1, deferred).

### 2.2 Loader semantics (fail-closed where it matters, friendly where it doesn't)

Every entry is **validated up front** (before any generator runs); a validation
failure is a config error (exit `3`), never a crash into a generic `1`.

| Situation | Disposition |
|-----------|-------------|
| File **absent** | "no derived docs registered" → `doc check` is clean (exit `0`). Mirrors slice-1's "empty repo passes". An adopter who never registers a doc is not in error. |
| File present, **not a mapping** / `derived_docs` not a list / an **entry that is not a mapping** (e.g. a bare string) / an entry missing `path` or `command` / `path` or `command` not a string | **malformed → exit `3`** (`EXIT_NO_CONFIG`, fail-closed; same class as "yaml corrupt"). A silently-skipped entry would drop a doc from the gate and mask real drift. (Validate as `isinstance(entry, dict)` *before* `entry.get(...)`, or a scalar entry raises `AttributeError`.) |
| `path` that **escapes the repo** — absolute (`/...`), or `..`-traversal whose `(repo_root / path).resolve()` is not under `repo_root` | **malformed → exit `3`**. Without this, `doc check --fix` would write outside the repo. This is a *sharper* edge than the `source-paths.yaml` loader it mirrors, because that loader only glob-*matches*; this one *writes*. |
| `command` empty / whitespace-only, or `shlex.split(command)` yields `[]` (no argv) | **malformed → exit `3`** (cannot run an empty argv; would otherwise raise on `subprocess.run([])`). |
| Duplicate `path` across entries (after repo-relative normalization) | **malformed → exit `3`** (ambiguous which generator owns the doc). |
| YAML parse error | **exit `3`**. |

A record/config error in the loader **dominates** and is reported before any
diff result (if the registry can't be parsed, the check set is unreliable) —
same precedence rule as slice-1 §4.1.

## 3. The generator contract & the regen-diff engine

### 3.1 Generator contract

A generator is **any command** (language-agnostic) that, when run from the repo
root, **prints the doc's canonical content to stdout and exits `0`**. That is the
entire contract. The tool owns everything else.

- **No-shell execution**: the tool splits `command` with `shlex.split` and runs the
  resulting **argv** directly (no `shell=True`) — no shell-metacharacter injection,
  no surprise globbing. **"Repo root"** (used for both `cwd` and the §2.2 path-escape
  containment check) is `find_harness_root(workspace)` — the `.harness`-containing
  directory, the same anchor every other command resolves against. This is a
  **deliberate divergence** from the existing `sensors/verification_runner.py`,
  which runs `shell=True` (it executes adopter-authored check commands that *want*
  shell features); doc-check generators only need to emit, so the hardened no-shell
  form is the right default here. (See §6's "why not reuse verification-runner".)
- **Bounded by a timeout**: `subprocess.run(argv, ..., timeout=_GENERATOR_TIMEOUT_S)`.
  This slice fixes the timeout as a **module constant** (`_GENERATOR_TIMEOUT_S = 30`);
  a per-entry / CLI override is **deferred** (§8) — there is no `timeout:` field in
  the §2.1 schema this slice, so the loader does not accept one. A generator that
  hangs (blocks on stdin, does network I/O) must **not** hang CI — `TimeoutExpired` →
  that doc is a **generator failure** (exit `4`, verdict unknown), never a hang.
  (This is a hazard slice-1 never had — it ran no subprocess.)
- **stdout handling**: capture stdout as **bytes**, then decode UTF-8. Generator
  stdout that is **not** valid UTF-8 → **generator failure** (exit `4`): the
  generator is broken. (Contrast the *on-disk* file in §3.2: undecodable bytes
  *there* are treated as drift, because `--fix` will overwrite them. Same byte
  problem, opposite routing, by design — generator-side = "tool broke", disk-side =
  "stale file to be regenerated".)
- **Trust model** (honest): `doc check` executes commands listed in a tracked
  `.harness/derived-docs.yaml` — the same trust surface as a `Makefile` or a CI
  step the repo owner wrote. Running them is expected (it is the repo's own CI
  generating the repo's own docs). The no-shell split limits the blast radius to
  "what the declared argv does", not arbitrary shell. (Bedrock ceiling unchanged:
  the owner already controls CI.)

### 3.2 The engine (tool-owned, uniform across all docs)

The diff/check/fix logic is **generic** — it is the difflib loop already proven in
`scripts/gen_cli_reference.py::run_check`, lifted up one level so every registered
doc shares it. The engine **evaluates every registered entry and aggregates** — it
never short-circuits on the first failure (a crash on doc B must not skip evaluating
doc A's drift, §3.3 M-rationale). Per entry the verdict is one of `in_sync` /
`drift` / `failed`:

1. Run `command` (bounded by the §3.1 timeout) → on non-zero exit / not-found /
   timeout / non-UTF-8 stdout → **`failed`** (record `command` + a short error).
2. Else `generated` = decoded stdout, **newline-normalized** (`\r\n` → `\n`) —
   see the determinism note below. Read the on-disk file at `path`:
   - missing / undecodable → **`drift`** (not a crash), as `gen_cli_reference
     --check` does today.
3. `generated == on_disk` → **`in_sync`**; else → **`drift`** (keep the unified
   diff).

**Determinism / why normalize newlines.** regen-diff compares byte-for-byte, so any
nondeterminism = perpetual false drift. The committed file is written by `--fix`
with `\n`; the generator's stdout is captured via subprocess, where platform/locale
newline translation can yield `\r\n`. Normalizing `\r\n`→`\n` on **both** sides
before compare is the **only** normalization applied (no whitespace/comment
folding — generators must be deterministic; if one isn't, that is the bug to fix,
not paper over). `render_markdown` is already deterministic (alphabetical
`sorted(...)` over commands, fixed trailing newline); `_TRANSITIONS` dict-literal
order is insertion-stable (3.7+); the §5.2 generator sorts its rows explicitly.

`--fix` writes `generated` to `path` (creating parent dirs) **only for `drift`
entries** — a `failed` entry is **never written** (writing partial/garbage stdout
from a crashed generator would corrupt a previously-good doc). `--fix` and the
plain check share the same generate-and-classify pass; only the post-classification
action differs (compare vs. write-on-drift).

**`--fix` exit code** (was undefined; pinned here): after writing, `--fix` **exits
`0` when it *resolved* the gate** — i.e. every entry is now `in_sync` because the
only problems were drift, which it just fixed. It does **not** re-report `2`: drift
that `--fix` repaired is no longer a violation, so `doc check --fix && git commit`
works as the local regen loop. But `--fix` cannot fix everything: if any entry was
`failed` it still exits `4`, and a malformed registry still exits `3` (those are not
drift). So `--fix` exit = `3` if registry malformed, else `4` if any generator
failed, else `0`. (It never returns `2` — that code means "unfixed drift exists",
which `--fix` by definition eliminated.)

### 3.3 Exit codes & output

Per the repo's global convention (`src/super_harness/exit_codes.py`):

| Code | Constant | Meaning |
|------|----------|---------|
| `0` | `EXIT_OK` | all registered docs in sync (or registry absent) |
| `2` | `EXIT_VALIDATION` | one or more docs **drifted** (the merge-blocking gate violation) |
| `3` | `EXIT_NO_CONFIG` | registry **malformed** (§2.2) |
| `4` | `EXIT_EXTERNAL_TOOL` | a generator **command failed** (non-zero exit, not found, timeout, or non-UTF-8 stdout) |

`1` (`EXIT_GENERIC`) is reserved for an uncaught internal failure. This mirrors the
sibling gates (`decision check`, `attest verify`): `0` pass / `2` block / `3`
no-or-corrupt config; `4` is added here for the new "external generator failed to
run" failure mode (slice-1 had no subprocess, so no `4`).

**Precedence picks only the *process exit code*; it never collapses signal.** All of
`2`/`3`/`4` are non-zero and **block the merge** — the exact winning code is
cosmetic for branch protection. When failures coexist the engine still evaluates
every entry (§3.2) and the **process exit code** is the most-severe present:
`3` (config — the whole set is unreliable) > `4` (a generator could not be
evaluated) > `2` (drift). The masking risk the reviewers flagged — "doc A drifts,
doc B's generator crashes, CI shows `4`, a human fixes the generator and misses the
drift" — is closed by the rule that **the `--json` body and the stderr summary
always list *every* bucket fully**, regardless of which code won. A human/CI must
read the buckets, not infer everything from the single code (all of which block
anyway).

`--json` emits the repo's **frozen 6-key envelope** (`cli/output.py::json_envelope`)
— NOT a bespoke shape (the `version` field is filled dynamically from
`__version__`; shown as `0.1.0` below only for illustration):

```json
{
  "command": "doc check",
  "version": "0.1.0",
  "status": "pass | fail",
  "exit_code": 0,
  "data": {
    "in_sync": ["docs/cli-reference.md"],
    "drift":   [{"path": "docs/state-machine.md", "diff": "<unified diff, truncated>"}],
    "failed":  [{"path": "...", "command": "...", "error": "..."}]
  },
  "errors": [{"code": "malformed_registry | path_escape | duplicate_path", "message": "...", "file": "..."}]
}
```

- `data.drift[].diff` is the unified diff **truncated to the first 40 lines** (with a
  trailing `... (N more lines; full diff on stderr)` sentinel when truncated; full
  diff always goes to stderr) — a large CLI/table change must not balloon the
  envelope to megabytes. The diff's `fromfile`/`tofile` labels are the
  **repo-relative `path`** and `path (regenerated)` (never an absolute or tmp path),
  so `--json` goldens are machine-stable.
- **Error-code vocabulary** (so golden tests can assert on `code`, not free-text):
  `errors[].code` ∈ { `malformed_registry` (covers unparseable-YAML / not-a-mapping /
  `derived_docs` not-a-list / non-mapping entry / missing-or-non-string `path`|`command` /
  empty-command), `path_escape` (absolute or `..`-escaping `path`), `duplicate_path`
  }. `data.failed[].error` is a short free-text reason (return code / `TimeoutExpired`
  / `not found` / `invalid UTF-8`); tests assert structurally on `{path, command}`
  and the failure *bucket*, **not** on the `error` string.
- Two **distinct renderers** (do not conflate): the **non-json** path prints
  `cli/errors.py::format_error` *text* to stderr; the **json** path emits the
  `errors[]` *dicts*. `errors[]` carries config/loader problems (the codes above);
  generator-run problems live in `data.failed`. The `file` key in `errors[]` is an
  intentional extension to the base `{code, message}` error object.

Status rollup (`output.py` semantics): `fail` if any drift, generator failure, or
registry error; else `pass`. `data.in_sync` / `data.drift` / `data.failed` each
sorted by path — deterministic for golden tests. (There is no `warning` tier —
unlike `decision check`'s dangling-down, every signal a derived doc produces is a
hard, blocking problem.)

## 4. CLI surface (new `doc` group)

- **`doc check [--fix] [--json]`** — the CI gate (§3). `--fix` regenerates each
  drifted doc in place; the json flag is the global `ctx.obj["json"]`
  (cli-command-surface §3.4).

That is the whole surface. No `doc list` / `doc gen` — `--fix` already covers
"regenerate", and the registry file *is* the list (a human reads the yaml). YAGNI:
add verbs only when a consumer needs them (§8).

## 5. Migrating `cli-reference` + adding `state-machine`

### 5.1 `gen_cli_reference` — exactly one mode: `--emit`

Ratified outcome (stated once, no hedging): `gen_cli_reference` **keeps only
`--emit`**; both its `--check` flag and its no-arg write mode are **removed**.
Regeneration is now `super-harness doc check --fix`.

- **`--emit`** prints `render_markdown(...)` to stdout (the function already returns
  the string — near-free). This is the registry command
  `python -m scripts.gen_cli_reference --emit`.
- **Removing the in-script `--check`** is the "two checkers for one doc can drift,
  so keep one" invariant slice-1 honored — `doc check` is the single check口.
- **Functions retained** (`render_markdown` / `run_check` / `write_reference`) so
  the existing `tests/unit/scripts/test_gen_cli_reference.py` (which calls the
  functions directly) keeps passing; only the `main()` arg-parsing surface changes.
- **`_HEADER_NOTICE` + module docstring must be updated**: the header notice is
  *baked into the committed `cli-reference.md`*, and today instructs "Regenerate
  with: `python -m scripts.gen_cli_reference`" — a now-removed command. It must
  point at `super-harness doc check --fix`, or the generated doc tells readers to run
  a dead command.
- **Migration ordering (or the gate red-fails on itself at birth)**: changing
  `_HEADER_NOTICE` changes `--emit` output, so the committed `cli-reference.md` is
  *instantly drifted* the moment the notice changes. Therefore the same commit that
  edits the notice MUST also **regenerate+commit `cli-reference.md`** (and commit the
  new `state-machine.md`) — otherwise `doc check` fails on this very repo the first
  time it runs. Also update the now-stale dead-command strings that survive in
  `run_check`'s stderr message and in `test_gen_cli_reference.py`'s remediation text
  (the functions are retained for the tests, but their "Run: `python -m
  scripts.gen_cli_reference`" hints point at the removed command).

### 5.2 New `scripts/gen_state_machine.py` — derive by *calling* the state machine

The universal transitions (`intent_declared` / `*_redeclared` / `intent_abandoned`
/ `implementation_restarted` / `implementation_invalidated` / informational no-ops)
are **imperative branches inside `compute_target_state`, not introspectable data** —
so a generator that *re-transcribed* them would carry its own copy that silently
drifts from the code (regen-diff would pass while the doc is wrong). The generator
therefore **does not read internal branches; it calls the real function**:

- Import `STATES` / `TERMINAL_STATES` from `super_harness.core.state` (the
  state-name SSOT), `KNOWN_EVENT_TYPES` (= `CORE_EVENT_TYPES | EXTENSION_EVENT_TYPES`)
  from `super_harness.core.events`, and `compute_target_state` / `INVALID` from
  `super_harness.core.transitions`. (Eager top-level imports — unlike
  `gen_cli_reference`, which lazy-imports the click tree inside `main()` to keep its
  test fixtures light; the state-machine generator has no such heaviness.)
- For every `(state, event)` in `(STATES ∪ {None}) × KNOWN_EVENT_TYPES`, call
  `compute_target_state(state, event)`. Because the rows are produced by *running*
  the code, the doc is genuinely `f(code)` — drift relative to the state machine is
  impossible-by-construction (it would require the function to return different
  values).
- **Two derived sections, both still computed by calling the function** (verified
  against the live code: the full grid yields 118 non-`INVALID` results, of which
  **70 are no-op self-loops** — informational events and `intent_declared` re-emit
  return `current` unchanged — leaving **~48 genuine transitions**). Emitting all
  118 would bury the real transitions under `X --event--> X` noise, so:
  1. **Transition table** = results where `to != from` (the ~48 real moves),
     columns `from` / `event` / `to`. The `(None, intent_declared)` row is a real
     transition (`from` is the start); render its `from` cell as `(start)`.
  2. **"Events that never change state"** = the event types for which
     `compute_target_state(s, e)` is `s` (or `INVALID`) for *every* state — derived
     by the same calls, listed as names. (This preserves the informational-event
     information without 70 redundant rows.)
- **Deterministic ordering** (regen-diff compares bytes — §3.2): sort the transition
  rows with an explicit, `None`-tolerant key —
  `key=lambda r: ("" if r.frm is None else r.frm, r.event)` (a bare `sorted()` over
  `None`-mixed `from` values raises `TypeError`); sort the no-op event list
  alphabetically.
- The state machine's code SSOT carries `@decision:d-fixed-transition-matrix`. There
  is **no machine-traceable doc→decision link** in this slice (that is §13.4,
  deferred) — the connection is prose, not a checked thread; do not over-read it.

### 5.3 `docs/getting-started.md` — link only (no prose rewrite this slice)

The recurring drift (commit `993fb92`, "stale 11-state prose" — note the prose said
*11* while `core/state.py` defines *10* states, i.e. narrative and code already
disagree) tempts a full de-duplication of getting-started's interwoven state
narrative. **That is deliberately out of this slice** (a reviewer flagged it as
scope creep, and it collides with the pending getting-started edits in OPEN-ITEMS
#6): rewriting hand-written narrative prose is prose-doc work, not the derivable
mechanism this slice ships, and risks dropping onboarding narrative that the
transition *table* does not encode.

So this slice does the **minimal, low-risk** thing: add `docs/state-machine.md`
(generated) and add a **one-line link** from getting-started to it ("for the full
state/transition reference see `state-machine.md`"). The actual de-duplication of
getting-started's stale state prose (and reconciling the 10-vs-11 discrepancy) is
registered as a **separate docs pass** in OPEN-ITEMS, to be done with the #6
backlog — not smuggled in here. (getting-started itself stays hand-written and
ungated; guarding *its* prose is the deferred teeth slice's job, not this one.)

### 6.1 Why not reuse `sensors/verification_runner.py`?

Umbrella §13.4 says super-harness should *orchestrate* contract sensors through the
existing `verification-runner` layers, not reinvent. That applies to **semantic
contract diffing** (oasdiff/buf/Atlas/ArchUnit) — none of which regenerates *this
repo's* doc from *this repo's* click tree; doc-leg is the **golden-file** rung the
umbrella §9 itself blesses, which has no mature external tool. But should it at
least ride on `verification-runner` rather than be a second subprocess runner? No,
and the reason is concrete: `verification_runner` runs a command that **self-asserts
pass/fail** (it owns its own verdict, `shell=True`, capture stdout/exit). doc-check
needs the tool to **own** the verdict — capture the generator's stdout, **diff** it
against the committed file, carry the **diff payload**, and **auto-fix** by writing
it back. The runner has no concept of diff, of a committed-baseline, or of `--fix`.
Forcing doc-check through it would mean each generator re-implements the diff (the
very non-uniformity model-A1 was chosen to avoid). So doc-check is a *distinct*
sensor with its own (hardened, no-shell) execution — a deliberate, argued
divergence, not an oversight.

### 6.2 What ships

What the **tool** ships:

1. **Adopter-facing**: a `doc check` step in the bundled `init` workflow template
   (`src/super_harness/templates/super_harness_workflow.yml`), so any repo that
   runs `init` gets it on `pull_request`. To make the dormant no-op **discoverable**
   (§2.2: an empty/absent registry passes green forever, with no signal it can be
   armed), `init` also writes a **skeleton `.harness/derived-docs.yaml`** containing
   only a commented example + a one-line "register docs your generator can re-emit"
   header — mirroring how `init` already ships a real `source-paths.yaml` default
   (`cli/init.py::_source_paths_default`). This turns the silent no-op into a
   self-documenting surface.
2. **Self-host**: this replaces the existing `cli-reference-drift` job. Mirror the
   sibling gate `decision-check`, which lives in its **own standalone workflow file**
   (`.github/workflows/decision-check.yml`) — *not* inline in `test.yml`. So: add a
   new `.github/workflows/doc-check.yml` (job/check name `doc-check`) running
   `super-harness doc check`, and **delete** the `cli-reference-drift` job from
   `test.yml`. **Required-check swap (footgun — must be atomic):** ruleset
   `17229037` currently requires `cli-reference-drift`; a deleted-but-still-required
   check **never reports → the branch is permanently un-mergeable**. So in the *same*
   change: **remove `cli-reference-drift` from the required set and add `doc-check`**
   (joining `decision-check` / `attest-verify` / `lint` / `e2e`). The conformance
   arm's rung-2 doc floor becomes hard, not advisory. **Behavioral delta to flag:**
   the old job blocked on `gen_cli_reference --check`'s **exit `1`** (drift);
   `doc check` blocks on **exit `2`** (`EXIT_VALIDATION`, aligning with the global
   convention and the sibling gates).
3. Documentation: an adopter must mark the check **required** via branch protection
   for it to block — the tool cannot make itself required on someone's repo
   (umbrella §7.3, bedrock ceiling).

Also update the repo's **green-invariant discipline** (the session/dev checklist):
`python -m scripts.gen_cli_reference --check` → `.venv/bin/super-harness doc check`.

## 7. Validation (this repo, as a generic instance)

Prove the slice end to end on a real repository:

- Register both docs; `doc check` reports both in sync (exit `0`).
- Hand-corrupt `docs/state-machine.md`; `doc check` reports drift (exit `2`) with a
  diff; `doc check --fix` restores it; re-check clean.
- A registry with a bad generator command surfaces exit `4`; a malformed
  `derived-docs.yaml` (incl. a `..`/absolute `path`, empty `command`, non-mapping
  entry) surfaces exit `3`.
- **Coexistence/precedence case**: a registry where doc A *drifts* AND doc B's
  generator *fails* → process exit is `4` (most severe), **and** the `--json` body
  populates both `data.drift` (A) and `data.failed` (B) — proving the engine
  evaluates every entry and never masks the drift behind the generator failure.
- `--fix` does **not** write a doc whose generator failed (B stays untouched).
- The CI step blocks a PR with a deliberately-drifted derived doc.

Enabling branch protection / marking the check required on this repo is a
validation step (subject to the account/token constraints in memory), not part of
the spec.

## 8. Deferred (each a separate functional scope) + honest limits

Registered in `private/OPEN-ITEMS.md` (umbrella item #7 / NEXT-SLICE doc-leg):

- **Prose half of the doc leg** — `@decision:` anchors inside docs. Two separable
  pieces (§1): the teeth-free *referential-integrity* check (real standalone value,
  deferred for sequencing) and the *change→re-review* proxy (needs the teeth). Taken
  together with the integrity-lock / teeth slice.
- **Decision-link on a derived doc** (umbrella §13.4) — add a `decision:` field +
  surface "this doc serves D" when a consumer reads it (additive, not omitted as
  worthless — §1).
- **`getting-started.md` state-prose de-duplication + 10-vs-11 reconciliation** —
  a docs pass, done with the OPEN-ITEMS #6 getting-started backlog (§5.3). This
  slice only adds the link.
- **`doc check [PATH]...` single-doc filter** — re-check/regen only the named
  registered docs, for the generator-iteration loop. Deferred until a registry grows
  past a few docs (at N=2 the friction is nil). No `doc list` — the yaml *is* the
  list.
- **Per-entry / CLI timeout override** — this slice fixes the generator timeout as a
  module constant (`_GENERATOR_TIMEOUT_S = 30`, §3.1); a `timeout:` registry field or
  `--timeout` flag is deferred until a real generator needs more than 30s.
- **Sedimentation arm** (umbrella §13.1 ARM-2) — different lifecycle, out of scope.

Honest limits of what this slice mechanically guarantees:

- It guards **derivable** docs only. A doc with no generator is not covered here
  (that is the prose leg). The hard, auto-fixable band is exactly "docs a
  deterministic generator can re-emit."
- `doc check` **runs declared commands** — it trusts the registry author (the repo
  owner), same as any CI config. No-shell argv limits, but does not eliminate, that
  trust.
- The CI rail is only as hard as branch protection (owner-controlled) — bedrock
  ceiling.
- A generator can be **wrong** (emit content that is itself stale relative to
  intent). regen-diff guarantees *doc == generator(code)*, not *generator(code) ==
  truth*. It welds the doc to the code; whether the code is right is the upstream
  conformance links' job.

## 9. One-line summary

> A `.harness/derived-docs.yaml` registry declares `doc → generator-command`; a
> single `doc check` regenerates each (no-shell, timeout-bounded), diffs against the
> committed copy, hard-fails on drift (exit `2`) and auto-fixes with `--fix`; the
> tool owns diff/fix/json/exit, the generator just emits to stdout. The registry
> earns its keep on **adopter extensibility**, not in-repo doc count. cli-reference
> is migrated in, and a `state-machine.md` is added that is derived by *calling* the
> live state machine over every state×event pair — total, impossible-to-drift,
> killing the recurring stale-state-prose class. Pure, mechanical, deterministic —
> the derivable (rung-2) half of the umbrella's doc leg; the prose half waits for
> the teeth.
